# Как я перевел 100 страниц технических чертежей с китайского на русский, сохранив каждую линию на своем месте

## Проблема

Пришел ко мне как-то чертеж китайского нагревателя. 4 страницы технической документации — допуски, размеры, материалы, требования. Всё на китайском. Нужно перевести на русский для производства.

Казалось бы — открываешь PDF в редакторе, заменяешь текст, готово. Но есть нюанс: **технические чертежи** — это вам не художественная литература. Там каждая подпись привязана к своей линии, размеру, детали. Сдвинешь на пиксель — и уже непонятно, к чему относится "Допуск ±0.1" — к этому отверстию или к соседнему.

Я начал искать готовые решения:

| Инструмент | Результат |
|---|---|
| Google Translate в PDF | Съехал весь layout, подписи поверх линий |
| Adobe Acrobat Pro ($25/мес) | Сносный layout, но плати каждый месяц |
| argos-translate (open source) | Качество перевода — "китайский->русский" через английский, теряется смысл |
| GPT API ($) | Хороший перевод, но layout разваливается |
| DeepL API ($) | Хороший перевод, но PRO-тариф для PDF |

Ни одно open-source решение не умело переводить **поколоночно** — каждая подпись на своем месте, с учетом поворота, размера шрифта, без белых прямоугольников.

Пришлось писать свое.

## Почему это сложно?

PDF — не HTML. Там нет понятия "абзац" или "строка". В технических чертежах текст может быть:

- Повернут на 90°, 180°, 270°
- Развернут вертикально (символы идут сверху вниз)
- Частью таблицы, где важно совпадение границ ячеек
- Наложен на линии чертежа — нельзя просто залить белым

Вот так выглядит структура одной страницы в PyMuPDF:

```
Block 0 (table header):
  └─ Line 0: "公差" → bbox=(450, 780, 500, 810), rotate=0
  └─ Line 1: "尺寸范围" → bbox=(450, 710, 530, 740), rotate=0

Block 1 (drawing note, rotated):
  └─ Line 0: "环保限制物质要求" → bbox=(470, 687, 560, 720), rotate=270
```

Каждый текстовый элемент — это не просто "строка". Это координаты (x0, y0, x1, y1), угол поворота, гарнитура шрифта, размер кегля, цвет. И все это должно **точно совпасть** в переводе.

## Архитектура

Я решил сделать pipeline из четырех стадий:

```
SENSE → FACT → LOGIC → CAUSALITY
```

### SENSE — парсинг PDF

Берем PyMuPDF (fitz), извлекаем все текстовые блоки. Важный нюанс: нужно различать **ротацию страницы** (page.rotation) и **ротацию отдельной строки** (dir / angle у линии). У меня была страница с rotation=270°, и все текстовые координаты были в "неповернутой" системе — текст физически лежал на y > height страницы, но из-за поворота был виден.

### FACT — подготовка

Фильтруем пустые строки, дедуплицируем, сортируем. Каждая строка получает уникальный ключ `(page, block_idx, line_idx)` — ровно одна транзакция на строку.

### LOGIC — перевод

Изначально я использовал argos-translate — open-source движок на C++. Проблема: для китайско-русской пары он переводил через английский как промежуточный язык (zh→en→ru). Терялись термины: "电热管固定片" превращалось в "heating tube fixing piece" вместо "крепежные пластины для нагревательных элементов".

Переписал на **DeepLX** — reverse-engineered API DeepL. DeepL дает лучшее качество для технических текстов, не требует API-ключа (бесплатный эндпоинт), поддерживает 30+ языков напрямую.

DeepLX стартует как локальный HTTP-сервер, я его поднимаю автоматически.

### CAUSALITY — наложение перевода

Самая сложная часть. Нужно:

1. **Удалить оригинальный текст**, не закрашивая чертеж. Решение: `page.add_redact_annot()`. Если передать `fill=None`, то область удаления не заливается цветом — линии и детали остаются видимыми.

2. **Вставить перевод в ту же позицию**. Использую `page.insert_text()` с параметром `rotate=`. Например, для строки с углом 270°:

```python
page.insert_text(
    fitz.Point(bbox[0], bbox[1]),
    "Крепежные пластины для нагревательных элементов",
    fontname="liberation-sans",
    fontsize=8.6,
    rotate=270
)
```

Проблема, с которой я столкнулся: русский текст часто длиннее китайского (китайский иероглиф обычно занимает меньше места, чем несколько латинских/кириллических букв). Текст вылезал за границу страницы. Пришлось добавить **overflow guard** — оценивать длину текста и сдвигать начальную точку, чтобы текст гарантированно оставался на странице.

## Аudit trail

Каждый перевод записывается в hash-chain audit лог:

```
hash_0 = sha256("")
hash_1 = sha256(hash_0 + "page:0 block:0 line:0 text:Допуск")
hash_2 = sha256(hash_1 + "page:0 block:0 line:1 text:Диапазон размеров")
...
```

Это позволяет верифицировать, что ни одна строка не была изменена после перевода. `--verify` флаг проверяет всю цепочку.

## P.S. Технические детали для интересующихся

**Почему не insert_textbox?** У PyMuPDF есть метод `insert_textbox()`, который автоматически переносит текст по границам прямоугольника. Звучит удобно, но на практике он:
- Ломает однострочные подписи (делает многострочными)
- Плохо работает с повернутым текстом
- Требует точного расчета высоты блока

Поэтому я использую `insert_text()` с явным указанием точки начала и ротации.

**Почему DeepLX, а не LibreTranslate?** LibreTranslate хорош, но:
- DeepL дает заметно лучшее качество для технических текстов
- DeepLX не требует регистрации или API-ключа
- Один бинарник на 24 MB — никаких npm install, Docker, Python зависимостей

**Совместимость шрифтов:** Для кириллицы в PDF нужны шрифты с кириллическими глифами. В Linux — Liberation Fonts (метрически совместимы с Times New Roman, Arial, Courier New). На Windows/Mac — встроенные шрифты. Если шрифт не найден, пайплайн падает с понятной ошибкой — лучше явно указать пользователю на проблему, чем получить "крякозябры" в PDF.
